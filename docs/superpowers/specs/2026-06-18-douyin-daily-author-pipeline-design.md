# 抖音博主每日新增视频接入现有 Workflow 设计

## 背景

当前仓库已经支持输入单个抖音视频链接后，依次执行以下流程：

1. 下载视频
2. 转换音频
3. 语音转文字
4. AI 总结
5. Git 提交

现需求是在保留现有单视频运行方式的前提下，新增“按天处理某个抖音博主当天新发布且未处理的全部视频”的能力，并提供定时任务自动执行。

目标博主主页：

`https://www.douyin.com/user/MS4wLjABAAAAWGs2N4r_PbCH8uXi07DlK8G5T-dz2EA_bnoWb00V5BaR_-LdVLMDxIfqFbU8qbwX`

## 目标

- 保留现有单视频入口：`python scripts/run_pipeline.py "<视频链接>"`
- 新增每日批处理入口，自动抓取目标博主当天发布的全部视频
- 仅处理未成功处理过的视频
- 使用 `video_id` 作为唯一去重键
- 接入现有 workflow，避免重复实现转文字、总结、提交逻辑
- 增加 GitHub Actions 定时任务，每天自动执行
- 支持手动触发当天批处理

## 非目标

- 不支持一次性回补全部历史视频
- 不支持本次实现内处理多个博主
- 不改写现有单视频下载、转写、总结的核心处理逻辑
- 不引入数据库，状态仍保存在仓库文件中

## 方案选择

### 方案 A：新增“博主页抓取脚本 + 状态文件 + 复用单视频流水线”

为每日批处理新增独立脚本，负责：

- 抓取目标博主当天视频列表
- 基于状态文件过滤已处理视频
- 对每个新增视频调用现有 `run_pipeline.py`

优点：

- 对现有功能侵入最小
- 单视频和批处理职责清晰
- 复用现有流水线，风险较低

缺点：

- 需要新增一层编排脚本

### 方案 B：直接扩展 `run_pipeline.py`

让 `run_pipeline.py` 同时支持单视频链接和博主主页 URL。

优点：

- 入口统一

缺点：

- 主脚本职责膨胀
- 主页抓取、批处理、单视频处理会耦合在一起

### 方案 C：完全独立的新批处理主流程

新建脚本重新编排下载、转写、总结、提交，不调用现有单视频入口。

优点：

- 批处理流程可以完全独立定制

缺点：

- 逻辑重复
- 后续维护成本更高

## 选型结论

采用方案 A。

原因：

- 最符合“接入已有 workflow，同时保留现有运行方式”的要求
- 可以最大限度复用现有脚本
- 出问题时更容易定位到“抓取层”还是“处理层”

## 架构设计

### 新增脚本

新增 `scripts/run_daily_author_pipeline.py`，作为每日批处理入口。

职责：

1. 加载目标博主主页 URL
2. 获取博主当天发布的视频列表
3. 读取已处理状态文件
4. 过滤出“当天发布且未处理”的视频
5. 逐条调用 `scripts/run_pipeline.py`
6. 仅在单条视频完整处理成功后写入已处理状态
7. 输出批处理结果并返回合适的退出码

### 新增模块

新增 `scripts/douyin_author_feed.py`，负责博主页抓取与解析。

职责：

- 根据博主主页 URL 获取视频列表
- 从返回结果中提取 `video_id`、视频链接、发布时间、标题
- 提供“按当天过滤”的统一逻辑

将抓取逻辑和批处理编排逻辑分开，避免 `run_daily_author_pipeline.py` 过重。

### 状态文件

新增状态文件：

`data/processed_douyin_videos.json`

建议结构：

```json
{
  "version": 1,
  "authors": {
    "MS4wLjABAAAAWGs2N4r_PbCH8uXi07DlK8G5T-dz2EA_bnoWb00V5BaR_-LdVLMDxIfqFbU8qbwX": {
      "videos": {
        "1234567890123456789": {
          "video_id": "1234567890123456789",
          "video_url": "https://www.douyin.com/video/1234567890123456789",
          "published_at": "2026-06-18T09:30:00+08:00",
          "processed_at": "2026-06-18T10:02:11+08:00"
        }
      }
    }
  }
}
```

这样设计的原因：

- 以 `video_id` 做唯一键，避免标题变化带来误判
- 预留 `authors` 结构，后续扩展多博主时不需要重做文件格式
- 保存 `published_at` 和 `processed_at`，便于排障和审计

## 数据流

每日批处理的数据流如下：

1. GitHub Actions 或手动命令启动 `run_daily_author_pipeline.py`
2. `run_daily_author_pipeline.py` 调用 `douyin_author_feed.py` 获取视频元数据
3. 脚本按上海时区筛选“当天发布”的视频
4. 脚本读取 `data/processed_douyin_videos.json`
5. 过滤掉已成功处理的 `video_id`
6. 对剩余视频逐条调用 `python scripts/run_pipeline.py "<视频链接>"`
7. 某条视频处理成功后，更新状态文件
8. 若存在失败视频，脚本最终以非 0 退出，便于定时任务暴露异常

## 当天判定规则

当天判定以 `Asia/Shanghai` 为准。

规则：

- 优先使用抓取结果中的标准发布时间字段
- 将发布时间解析为带时区时间
- 若其本地日期等于脚本运行日，则视为“当天发布”

降级策略：

- 如果抓取结果缺少可解析的发布时间，则该视频不纳入“当天自动处理”范围
- 这样做是为了避免误把旧视频当成当天视频处理

这个策略偏保守，但更符合“准确优先，禁止编造”的要求。

## 去重与补跑规则

去重规则：

- 以 `video_id` 作为唯一键
- 只有单条视频完整跑通现有 workflow 后，才写入状态文件

失败策略：

- 单条视频失败时，不写入状态文件
- 下一次定时任务或手动重跑时，会自动再次尝试该视频

补跑边界：

- 仅补跑“当天发布但尚未成功处理”的视频
- 默认不回补前一天及更早的视频

这样可以避免首次上线时把历史视频一次性全部扫入处理队列。

## CLI 设计

### 保留现有入口

```bash
python scripts/run_pipeline.py "https://www.douyin.com/video/xxxxx"
```

行为不变。

### 新增入口

```bash
python scripts/run_daily_author_pipeline.py
```

可选参数设计：

```bash
python scripts/run_daily_author_pipeline.py \
  --author-url "https://www.douyin.com/user/..." \
  --state-file "data/processed_douyin_videos.json"
```

默认情况下：

- `--author-url` 使用本次指定博主主页
- `--state-file` 使用仓库内默认状态文件

增加参数的原因是方便后续本地调试和扩展，但不改变默认使用方式。

## 错误处理

### 抓取失败

- 无法获取博主页数据时，本次批处理直接失败
- 不更新状态文件

### 单视频处理失败

- 记录失败的视频 ID 和链接
- 继续处理当天其他视频
- 全部处理结束后，只要有失败项，脚本以非 0 退出

### 无新视频

- 正常打印“当天无新增未处理视频”
- 脚本返回 0

### 状态文件损坏

- 读取失败时直接报错并退出
- 不自动覆盖或重置，避免误丢历史状态

## GitHub Actions 设计

新增一个 workflow，例如：

`.github/workflows/douyin_daily_author_pipeline.yml`

触发方式：

- `schedule`
- `workflow_dispatch`

行为：

1. checkout 仓库
2. 安装 Python 依赖
3. 执行 `python scripts/run_daily_author_pipeline.py`
4. 若处理后有新输出文件和状态文件变更，由现有脚本完成 Git 提交

设计原则：

- 定时批处理和现有单视频 workflow 并存
- 不删除现有手动运行能力

## 测试设计

本次实现至少补以下测试：

1. 当天过滤逻辑
   - 当天视频应被识别
   - 非当天视频应被过滤
   - 无法解析发布时间的视频应被过滤

2. 状态去重逻辑
   - 已处理 `video_id` 不应重复处理
   - 未处理 `video_id` 应进入执行队列

3. 成功后写状态
   - 单条成功后写入状态
   - 失败时不写入状态

4. 批处理结果汇总
   - 全成功返回 0
   - 部分失败返回非 0
   - 无新增视频返回 0

测试风格：

- 以纯函数和小模块为主，减少对真实网络和真实抖音页面的依赖
- 对外部命令调用采用可替换封装，便于测试

## 实现步骤

1. 提取或新增博主页抓取模块
2. 设计并实现状态文件读写模块
3. 编写当天过滤与去重逻辑测试
4. 实现 `run_daily_author_pipeline.py`
5. 接入 GitHub Actions 定时任务
6. 补充 README 使用说明
7. 完成本地验证

## 风险与约束

### 抖音页面结构变化

风险最高。博主页抓取高度依赖页面结构或接口字段，后续可能失效。

应对方式：

- 将抓取逻辑集中在单独模块，便于后续修复
- 保持日志足够清晰，失败时能快速定位是“抓不到列表”还是“解析不到发布时间”

### GitHub Actions 网络环境限制

抖音可能对 GitHub Actions 来源访问有限制，导致定时任务抓取失败。

应对方式：

- 代码上先按标准实现
- 若线上环境受限，再考虑迁移到本地 cron 或其他 Runner

### 现有 `run_pipeline.py` 的批处理适配

当前 `run_pipeline.py` 设计偏单次执行。批量逐条调用时，日志和失败传播需要明确。

应对方式：

- 新脚本只把它当黑盒命令调用
- 不在本次重构其主流程，降低改动面

## 验收标准

满足以下条件视为完成：

- 现有单视频命令仍可用
- 新脚本能识别目标博主当天新发布视频
- 已处理视频不会重复进入处理
- 失败视频不会被错误标记为已处理
- 可以通过 GitHub Actions 每天定时触发
- 当天无新视频时正常退出
- README 补充新的运行说明
